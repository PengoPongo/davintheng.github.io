My Role

What I Built

Lead Programmer & Chassis Co-Designer

I was responsible for the entire software side of this project — every screen, every interaction, every line of Python. I also helped design the physical chassis dimensions and layout that housed the Raspberry Pi, battery system, and speaker assembly.

This was my senior capstone project at UMass Lowell, built for Kin:pathic — a behavioral therapy company serving children with autism aged 18 months to 18 years. Their ask was simple but technically challenging: replace a traditional countdown timer with something a child could actually understand. Instead of numbers ticking down, a favorite image — a parent's face, a beloved character — slowly reveals itself as time passes.

My job was to make that happen in software, running on a Raspberry Pi with a touchscreen, a USB drive full of custom images, and a speaker that fires an alarm at zero. I built everything from scratch using Python and Pygame.

Code Deep Dive

How the Timer Animation Works

The visual centerpiece of the whole project is the countdown animation — a cyan "pie slice" that shrinks as time passes, uncovering the image beneath it. I wrote this using polar coordinates and Pygame's polygon renderer. Here's the exact function:

Screen_Codev3.py  ·  draw_timer_circle()
def draw_timer_circle(screen, center, radius, time_remaining, total_time): full_circle = 360 remaining_angle = (time_remaining / total_time) * full_circle # scale to degrees # Build a polygon from the center outward — a "pie slice" points = [center] # anchor at clock center for angle in range(0, int(remaining_angle) + 1): x = center[0] + radius * math.cos(math.radians(270 - angle)) y = center[1] + radius * math.sin(math.radians(270 - angle)) points.append((x, y)) pygame.draw.polygon(screen, BLUE, points)

The key insight: I start the sweep from 270° (the 12 o'clock position) and work clockwise. As time_remaining shrinks toward zero, remaining_angle shrinks too, so the polygon gets smaller and more of the image below it becomes visible. The image is drawn first, then the polygon is drawn on top — simple painter's algorithm.

Circular Image Masking

Getting images to display inside a circle — not a rectangle — required a custom masking function. Pygame doesn't have a built-in circular clip, so I built one using an RGBA surface and alpha blending:

Screen_Codev3.py  ·  crop_image_to_circle()
def crop_image_to_circle(image, radius): # RGBA surface so we get real transparency circle_surface = pygame.Surface((radius * 2, radius * 2), pygame.SRCALPHA) # White circle = the "keep" mask pygame.draw.circle(circle_surface, (255, 255, 255), (radius, radius), radius) # Blit the image with BLEND_RGBA_MIN — pixels outside the white # circle get their alpha zeroed out, making them transparent image_rect = image.get_rect(center=(radius, radius)) circle_surface.blit(image, image_rect, special_flags=pygame.BLEND_RGBA_MIN) return circle_surface

Clock Spokes with Trigonometry

The clock face ticks are drawn programmatically — no image files. I compute 60 spoke positions around the circle using polar math, making every 5th spoke longer to mark the hour positions, matching a real clock face:

Screen_Codev3.py  ·  draw_timer_spokes()
def draw_timer_spokes(color): center_x, center_y, radius = 400, 260, 195 even_spoke_length = 30 # long mark every 5 spokes odd_spoke_length = 10 # short marks in between for i in range(60): angle = math.radians(i * 6) # 360° / 60 = 6° per spoke spoke_length = even_spoke_length if i % 5 == 0 else odd_spoke_length start_x = center_x + radius * math.cos(angle) start_y = center_y + radius * math.sin(angle) end_x = center_x + (radius - spoke_length) * math.cos(angle) end_y = center_y + (radius - spoke_length) * math.sin(angle) pygame.draw.line(screen, color, (start_x, start_y), (end_x, end_y), 3)

Audio Alarm via GPIO PWM

When the timer reaches zero, I trigger an audio alarm through GPIO pin 18 using the gpiozero library's PWMOutputDevice. The function alternates between 440 Hz (A4) and 880 Hz (A5) tones for 5 seconds, driving the PAM8302A amplifier wired into the speaker in the lid:

Screen_Codev3.py  ·  play_alarm()
speaker = PWMOutputDevice(pin=18) def play_alarm(speaker): start_time = time.time() tones = [440, 880] # A4 and A5 while time.time() - start_time < 5: # plays for 5 seconds for freq in tones: speaker.frequency = freq speaker.value = 0.5 # 50% duty cycle = half volume sleep(0.2) speaker.off() sleep(0.1) speaker.off()

State Machine & Touch Input

The whole application runs on a three-state machine — HOME, IMAGE_SELECTION, and TIMER — managed in the main Pygame event loop. Every touch event is routed through collidepoint() checks against the active button rects for that state. This keeps the logic clean: each state only listens for the buttons that exist on its screen.

Screen_Codev3.py  ·  main event loop (condensed)
HOME, IMAGE_SELECTION, TIMER = 0, 1, 2 current_state = HOME while True: for event in pygame.event.get(): if event.type == pygame.MOUSEBUTTONDOWN: mouse_x, mouse_y = event.pos if current_state == HOME: if image_button.collidepoint(mouse_x, mouse_y): current_state = IMAGE_SELECTION loadImages() # scan USB drive on entry elif timer_button.collidepoint(mouse_x, mouse_y): current_state = TIMER elif current_state == TIMER: if play_button.collidepoint(mouse_x, mouse_y): timer_running = True elif reset_button.collidepoint(mouse_x, mouse_y): timer_running = False timer_minutes = time_set # restore user's chosen duration speaker.off() # Render the active screen if current_state == HOME: display_home_screen() elif current_state == IMAGE_SELECTION: display_image_selection_screen() elif current_state == TIMER: display_timer_screen() pygame.display.update() clock.tick(60)

Physical Design

Chassis & Enclosure

Beyond software, I was involved in designing the chassis layout. The enclosure needed to meet strict client requirements: no glass, non-porous exterior, cleanable with disinfectant wipes, no loose wires, and accessible internals with just a screwdriver. I worked through where every component would mount and how wiring would route, with the goal of making the inside as organized as the outside.

Layout decisions I made:

Pi + UPS HAT stacking

The UPS HAT sits directly on top of the GPIO headers, keeping the battery and power management directly under the Pi with minimal wiring.

Speaker in the lid

Mounting the speaker in the lid points audio forward toward the child rather than downward. The PAM8302A amp board is wired back to GPIO 18 on the Pi.

Side-panel power switch

The rocker switch is accessible to caregivers on the side of the device, out of reach when positioned on a shelf — meeting the classroom safety requirement.

Cable management

Ribbon cable for the touchscreen runs through the hinge channel. Power wires are zip-tied and clipped along the chassis walls, keeping everything secure.

Client requirements the chassis had to meet:

✓ No glass surfaces
✓ No latex materials
✓ Non-porous, wipe-clean exterior
✓ Accessible internals (screwdriver only)
✓ No loose wires or components
✓ Battery-powered, ≥ 8 hours runtime
✓ Non-toxic exterior materials

Problem Solving

Challenges I Worked Through

The Arduino couldn't handle it — so I switched to Raspberry Pi

We originally planned to use an Arduino Uno with a TFT LCD and SD card. After weeks of trying to optimize it, I hit a hard wall: the Arduino's limited processing power couldn't simultaneously manage graphics rendering and SD card image loading at any acceptable speed. I advocated for switching to a Raspberry Pi, which opened up Python, Pygame, and proper Linux file system access — but it also meant completely rewriting the software from scratch and sourcing a compatible touchscreen.

SD card → USB drive for image management

The original plan stored images on an SD card. In practice, caregivers found it awkward to remove, edit, and reinsert — especially with the Pi's own SD card already in use for the OS. I redesigned the image loading logic to look for a USB drive named "IMAGES" mounted under /media/, with a local folder fallback. This made swapping images as simple as plugging in a flash drive.

Shutdown safety with newer Raspberry Pi OS

The ATXRaspi shutdown board we used was designed for older Raspberry Pi OS versions that used init scripts. Newer OS versions manage shutdown through systemd, so the expected process was showing as inactive and the safe-shutdown button wasn't working. I traced the issue to the OS version difference and identified that a device tree overlay remapping the GPIO shutdown pin was the correct fix — something that would be implemented in a production revision.

Drawing a clock face without any image files

I wanted the timer's clock face — the 60 spokes, the circular outline — to be fully programmatic, with no image assets needed. I worked out the trigonometry to place 60 evenly-spaced tick marks around the circle, alternating long and short marks every 5 ticks to mirror a real clock. This also meant the clock renders perfectly at any resolution without needing texture scaling.

Outcome

Delivered & Working

The device was successfully delivered to Kin:pathic in January 2025. It met every client requirement — child-safe materials, 8+ hour battery, touchscreen interface, and fully customizable images via USB. The timer has been used in classroom settings to help children with autism manage transitions and understand the passage of time in a way that numbers never could.

Things I'd explore next given more time:

  • Fix the systemd-based shutdown via device tree overlay so the power button works reliably
  • Add a companion mobile app so caregivers can manage images wirelessly over Wi-Fi
  • Port the UI to a higher-resolution display and add smooth animation tweening
  • Explore a fully custom 3D-printed enclosure designed specifically for the final component layout
